# Story 0.1: Knowledge Store — Offline-First + Mappings Structure

**Status:** Done
**Prioridade:** P0 (bloqueante para Sprint 0)
**Esforço:** M
**Responsável:** @dev
**Epic:** Sprint 0 — Knowledge Store Foundation
**Origem:** Revisão arquitetural @architect (R2, R6)

## Objetivo
Garantir que o harvester opere com modo `offline` como padrão (scraping como opcional) e estruturar os diretórios de `mappings/` para separar artefatos gerados automaticamente de curações manuais, evitando sobrescrita em re-execuções.

## Contexto
O design original do harvester dependia de scraping externo (`documentation.sas.com`, `spark.apache.org`, `docs.databricks.com`) como modo primário. Esses sites mudam estrutura frequentemente, tornando o MVP não-determinístico. Adicionalmente, os `mappings/*.yaml` seriam gerados e aceitos curadoria manual no mesmo arquivo — qualquer re-harvest sobrescreveria as curações.

## Escopo
### Incluído
- Inverter prioridade no `knowledge/populate/harvester.py`: modo `offline` (HTMLs/PDFs locais em `./knowledge/raw_input/`) como padrão; `online` como flag explícita `--mode=online`
- Criar estrutura de diretórios:
  ```
  knowledge/mappings/
  ├── generated/    ← sobrescrito pelo build-mappings
  ├── curated/      ← nunca sobrescrito (curadoria manual)
  └── merged/       ← curated tem precedência sobre generated
  ```
- Implementar lógica de merge em `knowledge/store.py`: ao fazer lookup, consultar `merged/` com fallback para `generated/`
- Atualizar CLI `sas2dbx knowledge build-mappings` para gerar em `generated/` e depois fazer merge para `merged/`
- Documentar fluxo de curadoria em `knowledge/custom/README.md`

### Excluído
- Implementação dos harvesters específicos (sas_docs, spark_docs, dbx_docs) — esses ficam no escopo de outras tasks
- Seed dos YAML de mapeamento — tarefa separada
