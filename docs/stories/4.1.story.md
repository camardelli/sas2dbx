# Story 4.1: Generator — CellModel Intermediário para Notebooks .py e .ipynb

**Status:** Done
**Prioridade:** P1
**Esforço:** M
**Responsável:** @dev
**Epic:** Sprint 4 — Generation
**Origem:** Revisão arquitetural @architect (R10 — decisão arquitetural pendente)

## Objetivo
Implementar `generate/notebook.py` com um modelo de célula intermediário (`CellModel`) agnóstico de formato de saída, e renderizadores separados para `.py` (Databricks source format com `# MAGIC` comments) e `.ipynb` (Jupyter notebook com metadata Databricks). Isso evita duplicação de lógica e suporta ambos os formatos sem acoplamento.

## Contexto
O `sas2dbx.yaml` define `notebook_format: py | ipynb`, mas o notebook já estava se tornando um Databricks-específico. Com Databricks migrando cada vez mais para Repos e `.ipynb` nativo (e a maioria dos clientes com preferências diferentes), gerar os dois formatos a partir de um modelo único é a abordagem correta. Sem `CellModel`, qualquer lógica de ordenação ou rastreabilidade de células teria que ser implementada duas vezes.

## Escopo
### Incluído
- `CellModel` dataclass em `generate/notebook.py`:
  ```python
  @dataclass
  class Cell:
      cell_type: CellType          # CODE | MARKDOWN
      source: str                  # conteúdo da célula
      sas_origin: SASOrigin | None # rastreabilidade: job, linha início/fim, construto
      order: int                   # posição na sequência

  @dataclass
  class CellModel:
      job_name: str
      sas_source_file: str
      migration_date: str
      confidence: float
      warnings: list[str]
      cells: list[Cell]
  ```
- `NotebookRenderer` interface base:
  ```python
  class NotebookRenderer(ABC):
      @abstractmethod
      def render(self, model: CellModel, output_path: Path) -> None: ...
  ```
- `DatabricksPyRenderer` — renderiza para `.py` com `# MAGIC %md` e `# COMMAND ----------`
- `JupyterIpynbRenderer` — renderiza para `.ipynb` (JSON) com metadata Databricks (`"metadata": {"application/vnd.databricks.v1+notebook": {...}}`)
- `NotebookGenerator` seleciona renderer pelo `notebook_format` em config
- Célula de header gerada automaticamente (metadata: job, confiança, warnings)
- Célula de imports gerada automaticamente (F, Window, etc. detectados no código)
- `SASOrigin` com rastreabilidade (`job_name`, `start_line`, `end_line`, `construct_type`)

### Excluído
- Execução dos notebooks
- Validação de formato pelo Databricks
- Suporte a outros formatos além de `.py` e `.ipynb`
