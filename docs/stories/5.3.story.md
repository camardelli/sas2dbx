# Story 5.3: Pipeline Integration — Inventory Before Engine + Preflight Refactor

**Status:** Draft
**Sprint:** 5 — Inventory Foundation
**Depende de:** 5.2 (Source Inventory Module)
**Bloqueada por:** 5.2

---

## Contexto

Com o inventory module funcionando standalone (Story 5.2), esta story o integra no pipeline de migração. O worker passa a chamar inventory ANTES do engine, o schemas.yaml é gerado antes da primeira chamada LLM, e o preflight perde o ownership de schemas.yaml (não sobrescreve se já existe).

## Objetivo

Integrar `sas2dbx/inventory/` no pipeline principal: worker → inventory → gate → engine. Preflight refatorado para não sobrescrever schemas.yaml existente. CLI recebe `--inventory-mode`.

## Acceptance Criteria

- [ ] Engine recebe schemas pré-populados na primeira transpilação (não só na re-transpilação)
- [ ] Preflight não sobrescreve schemas.yaml se arquivo já existe (inventory escreveu)
- [ ] `--inventory-mode=STRICT` (default) bloqueia se SOURCE ausente
- [ ] `--inventory-mode=FORCE` prossegue com warning
- [ ] `--inventory-mode=SKIP` desabilita inventory (comportamento pré-5.2)
- [ ] Job 107 transpila sem erros de coluna na primeira passagem (schemas disponíveis antes do LLM)
- [ ] `pytest` 100% verde

## Fluxo após integração

```
scan_sas_files()
  ↓
split_blocks() → SASBlock[]
  ↓
InventoryExtractor.extract()
  ↓
DatabricksChecker.check_all()
  ↓
InventoryGate.evaluate()  →  BLOCK? → abort, salva inventory_report.json
  ↓
InventoryReporter.save_schemas(output_dir/schemas.yaml)   ← ANTES do engine
  ↓
TranspilationEngine(schemas_pre_populated=True)
  ↓
Preflight(skip_schema_write=True se schemas.yaml existe)
```
